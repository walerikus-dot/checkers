01

Игровая логика и правила

Поддерживаемые варианты правил

Рекомендуется
Русские шашки
  • Доска 8×8, 12 шашек на сторону
  • Обязательное взятие
  • Дамка ходит на любое расстояние
  • Множественные взятия разрешены
Международные (10×10)
  • Доска 10×10, 20 шашек на сторону
  • Обязательное взятие максимального кол-ва
  • Дамка — дальний ход по диагонали
  • Правило 25/50 ходов без взятия
Английские (Checkers)
  • Доска 8×8, 12 шашек на сторону
  • Взятие не обязательно
  • Дамка — только 1 шаг по диагонали
  • Ход только вперёд до получения дамки
Кастомные правила
  • Конфигурируемые параметры через JSON
  • Гибкий размер доски
  • Настройка любых механик
  • Опционально, V2+
02

Настройки лобби

⚙️ Хост задаёт параметры
  • Тип правил — русские / международные / английские
  • Размер доски — 8×8 или 10×10
  • Обязательность взятия — вкл/выкл
  • Множественные взятия — вкл/выкл
  • Правила дамки — стандартные / расширенные
  • Таймер — блиц (3+2, 5+3) / кастом / без ограничений
👤 Экран подтверждения игрока 2
  • Превью всех настроек игры перед входом
  • Тип правил и размер доски
  • Имя хоста и его текущий рейтинг
  • Время на ход / тип таймера
  • Кнопка подтверждения «Войти в игру»
  • Collapsible блок с правилами во время партии
03

AI-противник

Лёгкий
глубина: 1
  • Случайные допустимые ходы
  • Соблюдение базовых правил
  • Без оценки позиции
Средний
глубина: 2–3
  • Minimax
  • Простая оценка позиции
  • Приоритет взятий
Сложный
глубина: 4–6
  • Minimax + Alpha-Beta pruning
  • Контроль центра доски
  • Эвристика по дамкам
Супер сложный
глубина: 6–8 · опционально
  • Transposition table
  • Opening book (дебюты)
  • Динамическая глубина поиска
→
Рекомендация: Начать с лёгкого и среднего уровней (Minimax без Alpha-Beta). Оптимизацию и transposition table добавлять итеративно после стабилизации ядра игры.

Сервер хранит и валидирует

  • Аккаунты и профили пользователей
  • Рейтинг и история партий
  • Состояние активных игр
  • Турниры и результаты матчей
  • Очереди matchmaking (Redis)
  • Каждый ход подтверждается сервером

Клиент обрабатывает локально

  • Рендер доски (Canvas или DOM)
  • AI-логика для игры против компьютера
  • Предварительная валидация ходов (для UX)
  • Анимации и звуковые эффекты
  • Offline-кеш (PWA Service Worker)
→
Рекомендация: Turn-based + WebSocket как стартовая модель. Сервер подтверждает каждый ход — проще реализовать, проще отлаживать, меньше нагрузки по сравнению с real-time.
05

Matchmaking

Быстрая игра
  • Случайный соперник из очереди
  • Фильтр по типу правил
  • Диапазон рейтинга ±150 Glicko-2
  • Очередь через Redis pub/sub + TTL
  • Расширение диапазона со временем ожидания
Приватная игра
  • Создание лобби с уникальным кодом
  • Инвайт-ссылка для друга
  • Полная настройка параметров хостом
  • Экран подтверждения для второго игрока
06

Турниры

Форматы

  • Single Elimination — на выбывание
  • Double Elimination — с нижней сеткой
  • Round Robin — каждый против каждого
  • Swiss — опционально, V3+

Управление (администратор)

  • Включение / отключение типов турниров
  • Авто-создание или ручная модерация
  • Сетка турнира с расписанием
  • Автоматическое продвижение победителей
  • Призовые места и рейтинговые очки

Рейтинговая система

📊 Glicko-2
  • Рейтинг (r) — начальное значение 1500
  • Отклонение (RD) — неопределённость рейтинга
  • Волатильность (σ) — стабильность результатов
  • Пересчёт после каждого рейтингового периода
  • Увеличение RD при длительном бездействии
  • Корректная оценка новых игроков
→
Рекомендация: Glicko-2 предпочтительнее Elo — лучше обрабатывает неактивных игроков и даёт более точную оценку после небольшого числа партий.

Деплой (Ubuntu 24)

Технологический стек

Node.js / NestJS PostgreSQL Redis Nginx Docker + Compose Let's Encrypt (SSL)

Процесс деплоя

  • Установка Docker + Docker Compose
  • Контейнеры: backend, db, redis, nginx
  • Nginx как reverse proxy (порт 80/443)
  • SSL через Certbot + Let's Encrypt
  • Автозапуск через systemd / docker restart policy
  • GitHub Actions для CI/CD (опционально)
12

Безопасность

🔑
JWT авторизация
Access + refresh токены. Хранение refresh в httpOnly cookie.
✅
Валидация ходов
Каждый ход проверяется на сервере. Клиент не является источником истины.
🚦
Rate limiting
Ограничение запросов по IP и по аккаунту. Redis-счётчики.
📋
Логирование
Все ходы записываются. Аудит подозрительных паттернов.
🛡️
Input validation
class-validator + Pipes в NestJS. Sanitize всех входных данных.
🔒
HTTPS / WSS
Только зашифрованные соединения. HTTP → HTTPS редирект через Nginx.
